# Contributing

Thank you for helping improve Video Publish Skill. Small, focused pull requests are easiest to review.

## Before opening an issue

- Search existing issues first.
- Use the structured Bug or Feature template.
- Redact account names, phone numbers, QR codes, cookies, tokens, local absolute paths, unpublished
  media, and creator-page screenshots containing private data.
- Use the private process in [SECURITY.md](SECURITY.md) for vulnerabilities or publish-guard bypasses.

## Development setup

```bash
corepack enable
pnpm install --frozen-lockfile
pnpm audit:dependencies
pnpm check:opensource
pnpm fmt:check
pnpm lint
pnpm check-types
pnpm build
```

Node.js 22.13+ and pnpm 11 are required for repository development. The bundled Skill CLI continues
to target Node.js 20. FFmpeg, Python/faster-whisper, Chrome, and Ego Lite are only
needed when manually exercising the corresponding runtime paths.

## Repository boundaries

- `packages/core/src/` owns schemas and platform-independent contracts.
- `packages/runtime/src/` owns deterministic local CLI behavior.
- `skills/prepare-video-publish/` is the installable Codex Skill.
- `vendor/oil-video-publisher/` is a pinned upstream snapshot; do not silently rewrite its history.
- `skills/prepare-video-publish/scripts/video-publish.mjs` and the bundled `ego-publisher/` directory
  are generated by `pnpm build` and must remain in sync with source and vendor inputs.

Do not commit `.video-publish/`, browser logs, videos, credentials, account data, or exported login
state. Never weaken the final-publish guard to make an adapter easier to implement.

## Changing platform adapters

Platform DOMs and labels change frequently. A platform change must:

1. inspect current page facts before mutation;
2. distinguish login, upload, editor, dialog, and ready states;
3. mutate only the requested draft;
4. verify exact metadata entities and cover receipts after mutation;
5. preserve idempotency and interruption recovery;
6. keep the final publish action blocked;
7. document any unsupported or partially verified capability.

Do not include real user sessions or production account evidence in a pull request. Use synthetic or
fully redacted fixtures when maintainers explicitly request test material.

## Vendored upstream updates

When updating `oil-oil/video-publisher-skill`:

1. record the new full commit SHA in `vendor/oil-video-publisher/UPSTREAM.md` and
   `THIRD_PARTY_NOTICES.md`;
2. preserve the upstream MIT license;
3. describe local patches and conflict resolutions;
4. rebuild the installable Skill;
5. review the generated diff, especially final-publish guards and login-state handling.

## Pull requests

- Use a Conventional Commit-style title such as `fix(runtime): preserve exact topic entities`.
- Explain the user-visible behavior, safety impact, verification performed, and documentation changes.
- Update `CHANGELOG.md` for user-visible changes.
- Run all quality commands above and include relevant output in the PR.
- Keep generated files in the same commit as their source changes.
